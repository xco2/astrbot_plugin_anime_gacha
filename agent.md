# agent.md

本文件用于说明 `astrbot_plugin_anime_gacha` 插件的结构、职责和修改约定，方便后续代理/协作者快速接手。

## 插件定位

这是 AstrBot 的一个动漫信息插件，当前主要提供 3 类能力：

1. 番剧推荐：`/抽番`
2. 新番查询：`/今日新番`、`/更新番剧数据`、`/检查番剧数据`
3. 萌娘百科检索：`/萌娘搜索`，以及提供给 LLM 的工具 `search_moegirl`

插件入口位于 `main.py`，核心数据管理位于 `data_holder.py`。

## 当前目录结构

- `main.py`：AstrBot 插件入口，注册命令、组织返回消息、调用 `DataHolder` 与萌娘百科检索逻辑。
- `data_holder.py`：本地缓存、季度数据更新、推荐番剧缓存、图数据构建与番剧查询核心逻辑。
- `anime_scraper/scraper.py`：**抓取路由层**。按季度把请求分发到不同世代的抓取器，并提供批量回归测试入口。
- `anime_scraper/scraper_2601.py`：**旧版/历史页面抓取器**，面向 `202604` 之前的页面结构。
- `anime_scraper/scraper_2604.py`：**新版页面抓取器**，面向 `202604` 及之后的新页面结构。
- `anime_scraper/moegirl_scraper.py`：萌娘百科搜索与页面内容获取。
- `split_long_text.py`：长文本拆分工具。
- `anime_datas/`：本地缓存目录，保存番剧数据、推荐数据、图谱数据等运行产物。
- `metadata.yaml`：AstrBot 插件元信息。
- `README.md`：插件功能说明与用户指令文档。
- `requirements.txt`：插件额外依赖。
- `graph_data_viwer.py`、`graph.html`：图谱/调试相关辅助文件。

## 已确认的主要功能

### 1. `/抽番`
- 从推荐番剧源获取数据。
- 随机挑选 12 部番剧。
- 生成 HTML 卡片并渲染为图片，再与文字列表一起发送。

### 2. `/今日新番`
- 查询当前季度今日更新的番剧。
- 每 12 小时允许触发一次联网更新。
- 若临近下一季度，也会附带下一季度同 weekday 的更新信息。
- 会把 UTC+8 时间转换成 UTC+9 展示。

### 3. `/更新番剧数据 YYYYMM`
- 手动更新指定季度的番剧数据。
- 输入会被归一化到季度起始月份：`01/04/07/10`。

### 4. `/查番 名称`
- 查询番剧详细信息。
- 展示日文名、档期、更新时间、总集数、类型、CV、标签、Staff 等。
- 若存在多个匹配项，会附加相似番名列表。

### 5. `/检查番剧数据`
- 检查本地已有季度数据。
- 根据当前时间反推理论上应覆盖的季度，并列出缺失项。

### 6. `/萌娘搜索 问题` 与 `search_moegirl`
- 命令模式：先结合对话上下文，让 LLM 提取搜索关键词，再调用萌娘百科搜索。
- 工具模式：作为 `llm_tool` 暴露给模型，供问答时检索动漫/人物/作品资料。

## 数据与运行机制

### `DataHolder` 负责
- 维护本地 JSON 缓存。
- 管理今日推荐缓存。
- 管理 RDF 图谱与番剧名称索引。
- 按季度抓取、存储、查询番剧数据。
- 限制抓取频率，避免短时间过快请求。

### 当前抓取架构（已更新）

当前 `anime_scraper` 已改为**按页面世代分文件维护**：

1. `anime_scraper/scraper.py`
   - 作为统一入口与路由层。
   - 对外继续暴露：
     - `download_new_anime_datas(schedule_time: str)`
     - `get_today_recommend()`
   - 内部根据季度决定调用哪个版本的抓取器。

2. `anime_scraper/scraper_2601.py`
   - 保留旧版抓取逻辑，主要兼容 `202604` 之前的历史页面。
   - 已增加少量兜底，避免部分历史季度因为块识别失败而直接抛异常。

3. `anime_scraper/scraper_2604.py`
   - 保存为适配 `202604` 新页面结构的新抓取器。
   - 已修复新版页面的：
     - 日更时间表提取
     - 误把 `120px` 日更小卡片当作详情表的问题
     - `unknown_type` 的误报问题

### 当前路由规则
- `schedule_time >= "202604"`：走 `scraper_2604.py`
- `schedule_time < "202604"`：走 `scraper_2601.py`

### 本地回归测试入口
- `anime_scraper/scraper.py` 现在带有批量回归测试入口。
- 可直接批量测试从 `201910` 到 `202604` 的固定季度列表。
- 输出包括：
  - `route`
  - `daily`
  - `anime`
  - `details`
  - `sample_day`
  - `sample_anime`
  - `sample_detail`
  - `summary`

### 当前验证结论
- 分代路由已生效：
  - `202604` 走新版抓取器
  - 更早季度走旧版抓取器
- 最新版 `202604` 页面当前已验证：
  - `daily_count = 7`
  - `anime_count = 69`
  - `detail_count = 70`
  - `unknown_type = 0`
- 历史季度中仍有部分 `daily=0` 的 warning，但已确认其中多数是**页面本身没有独立时间表**，不属于当前代码故障。

### 本地缓存文件
运行时会在 `anime_datas/` 下生成或维护：
- `anime_datas.json`
- `today_recommend_anime.json`
- `anime_graph.ttl`
- `all_anime_names.txt`

修改代码时要区分：
- **源码文件**：应纳入版本控制
- **缓存/运行产物**：通常不应随意改动其格式，更不要把测试过程中临时生成的数据误当作源码逻辑提交

## 修改约定

在这个插件目录下工作时，优先遵循以下约定：

1. 保持改动聚焦，不顺手重构无关逻辑。
2. 修改命令行为时，同步检查 `README.md` 与 `metadata.yaml` 是否需要更新。
3. 优先修根因，不做只覆盖表面的补丁。
4. 新增代码尽量沿用现有风格，避免无意义重命名。
5. 若涉及路径处理，优先考虑 `pathlib.Path`；但本插件当前大量使用 `os.path`，除非明确重构，否则不要大范围混改。
6. 不要把运行缓存、抓取结果、临时分析文件当作源码文档提交。
7. 除非任务明确要求，不要改动 `.git/`、`.idea/` 或调试展示文件。
8. 新增注释尽量使用英文；现有中文注释保留即可，不必为风格统一而大规模改写。

## 风险点与注意事项

1. `main.py` 当前同时承担：命令入口、展示拼装、部分搜索流程控制，职责偏多；修改时要小心避免牵一发动全身。
2. 萌娘百科搜索依赖外部网页/API，可能受风控、关键词质量或网络波动影响。
3. 番剧数据更新依赖外部抓取源，失败时应优先保证命令能优雅报错。
4. 插件中存在基于当前时间、季度切换、weekday 的逻辑，改动时要留意边界条件。
5. 图谱与缓存文件之间存在版本关联，若调整数据结构，需要同时考虑版本兼容与重建逻辑。
6. `yuc.wiki` 的季度页面**不是统一模板**，很可能是按季度手工维护；不要再假设一个抓取器能稳定覆盖全部年代页面。
7. 以后如果 `yuc.wiki` 再次改版，优先新增新的 `scraper_YYYYMM.py`，并在 `anime_scraper/scraper.py` 里扩展路由，而不是直接把所有兼容分支堆回单文件。
8. 新版页面中存在 `120px` 的日更小卡片与 `500px` 的详情大表并存的情况，修改新版抓取器时要特别小心不要把小卡片误识别进 `anime_details`。

## 建议的后续维护方式

若后续继续开发，推荐按下面思路进行：

1. 将 `main.py` 中的展示模板、命令逻辑、萌娘搜索流程逐步拆分。
2. 为 `data_holder.py` 的季度计算、时间转换、数据缺失检查补充可验证测试。
3. 把外部抓取失败、空结果、格式异常的处理统一起来。
4. 明确 `anime_datas/` 下哪些文件属于可重建缓存，哪些属于兼容性关键数据。

## 本文件用途

后续代理在修改本插件前，应先阅读本文件，再阅读：
- `main.py`
- `data_holder.py`
- `README.md`
- `metadata.yaml`

这样可以更快判断某次需求属于：命令层修改、数据层修改、搜索层修改，还是文档/元信息同步修改。
